## Что это за проект

Telegram-бот для контроля персонала через фоточек-листы.
Сотрудники проходят чек-листы, подтверждая каждый пункт фотографией.
ИИ проверяет фото на соответствие задаче. Данные уходят в Google Sheets.

---

## Главный приоритет

- Быстрый стабильный MVP без переусложнения
- Не добавлять функционал которого нет в требованиях
- Это чек-лист бот, не CRM и не платформа

---

## Стек

- Node.js + TypeScript (strict)
- Telegraf (Telegram Bot API)
- SQLite через Prisma (MVP)
- S3-совместимое хранилище (Cloudflare R2 / Yandex Object Storage)
- Google Drive — опционально, только если включён флагом ENABLE_GOOGLE_DRIVE=true
- Google Sheets через Make webhook (HTTP POST)
- OpenAI gpt-4o-mini для проверки фото
- Sharp для наложения водяного штампа на фото

---

## Границы MVP (не выходить за них)

1. 7 должностей — берутся из src/config/checklists.json
2. 3 типа чек-листов: open, periodic, close
3. Каждый пункт чек-листа требует фото
4. Один активный чек-лист одного типа на пользователя
5. Состояние хранится в БД, не в памяти
6. Админ-панель как UI — НЕ входит в MVP
7. Управление чек-листами — через конфиг (JSON) или импорт
8. Команда /reload — только для администраторов (из ADMIN_IDS)

## Логика FSM

```
/start
  → не зарегистрирован: запрос имени → выбор должности
  → зарегистрирован: меню чек-листов по роли и времени

Старт чек-листа:
  → создать run
  → показать первый вопрос

Получено фото:
  → наложить штамп (имя + дата + время) через Sharp
  → загрузить в S3
  → AI-проверка если задано ai_rule
  → сохранить verdict в answers
  → POST в Make webhook
  → следующий вопрос

Завершение:
  → закрыть run
  → если тип close → рассчитать смену (shifts)
```

## Расчёт смены

- Начало смены = время первого фото с verdict "ok" по правилу uniform_check в чек-листе open
- Конец смены = время завершения чек-листа close
- Итог записывается в shifts: minutes (отработано), fail_count (количество неверных фото)
- fail_count влияет на расчёт зарплаты (формула на стороне Google Sheets)
- Ежемесячная статистика суммируется в отдельном листе таблицы

---

## Расписание чек-листов

- open — отправляется в начале смены (время из конфига)
- close — отправляется в конце смены (время из конфига)
- periodic — отправляется каждые 2 или 3 часа (интервал из конфига)
- Расписание реализовать через node-cron
- Если пользователь не прошёл чек-лист в отведённое время — фиксировать пропуск

## Одноразовые коды (опционально, если запрошено клиентом)

- Перед каждым фото бот генерирует одноразовый код (например: A7X4)
- Код действителен 2-3 минуты
- Сотрудник пишет код на бумаге и фотографирует вместе с объектом задачи
- ИИ проверяет наличие кода на фото
- Если время истекло — фото отклоняется автоматически
- Реализовывать ТОЛЬКО если клиент явно подтвердил этот вариант

## Обработка фото

- Принимать message.photo и document (если изображение)
- Скачивать через Telegram getFile
- Накладывать водяной штамп: имя сотрудника + дата + время (библиотека Sharp)
- Загружать в S3, возвращать публичный URL
- В dev-режиме допускается локальное хранение
- Галерею заблокировать нельзя на уровне Telegram Bot API — это ограничение платформы
- Защита от старых фото реализуется через: штамп + AI-проверка актуальности + одноразовые коды (если включены)

## AI-проверка

Сервис aiService с абстракцией провайдера (OpenAI / GigaChat).

Возвращать строго:
```json
{
  "verdict": "ok" | "fail",
  "reason": "string",
  "confidence": 0.0 - 1.0
}
```

Правила MVP: uniform_check, steam_check.
Никогда не заявлять 100% точность. Хранить как "оценку ИИ".

## Безопасность

### Доступ и авторизация
- Все команды администратора (/reload и др.) — проверять telegram_id через ADMIN_IDS из .env
- Пользователь может работать только со своими runs и answers — никогда не давать доступ к чужим данным
- Незарегистрированный пользователь не получает доступ к чек-листам

### Секреты и переменные окружения
- Все ключи (BOT_TOKEN, OPENAI_API_KEY, S3_SECRET_KEY и др.) — только через .env
- .env никогда не коммитить в git — добавить в .gitignore
- В коде не хардкодить никакие токены, ключи, URL вебхуков

### Валидация входящих данных
- Проверять что входящий файл является изображением перед обработкой (validateImage.ts)
- Игнорировать сообщения не того типа (текст вместо фото и наоборот)
- Ограничить размер принимаемого файла (максимум 20 МБ — лимит Telegram)

### Защита от злоупотреблений
- Один активный run одного типа на пользователя — не давать запустить второй
- Не принимать фото вне контекста активного вопроса
- Логировать все подозрительные действия (попытки вызвать чужие данные, неверные команды)

### Хранение данных
- Фото хранить в S3 с настройкой доступа согласно конфигу бакета
- Не хранить персональные данные сотрудников сверх необходимого (имя + telegram_id + роль)
- БД бэкапить регулярно — особенно важно для shifts (данные о зарплате)

## Защита от намеренного саботажа сотрудниками

Сотрудник не должен иметь возможность "сломать" бота чтобы избежать фотофиксации.

### Спам и флуд
- Текстовый спам в момент ожидания фото — игнорировать, отвечать коротко: "Пожалуйста, отправьте фото"
- Не падать и не зависать от любого количества входящих сообщений
- Обрабатывать все входящие через try/catch — любая ошибка логируется, бот продолжает работу

### Попытки сбросить состояние
- /start в середине чек-листа — НЕ сбрасывать активный run, показать текущий вопрос
- Повторный /start — показать что чек-лист уже в процессе и предложить продолжить
- Нельзя отменить начатый чек-лист без участия администратора

### Отправка неверных файлов
- Документ не являющийся изображением — отклонить: "Отправьте фото, не файл"
- Видео, стикеры, голосовые сообщения — игнорировать
- Слишком маленькое фото (менее 100x100px) — отклонить, попросить нормальное фото

### Блокировка бота
- Если сотрудник заблокировал бота — фиксировать пропуск чек-листа автоматически
- Ошибка "bot was blocked by the user" — обрабатывать gracefully, не падать
- Разблокировка бота — сотрудник продолжает с текущего активного вопроса

### Манипуляции со временем
- Все временные метки фиксировать на стороне СЕРВЕРА, не доверять времени от клиента
- Время штампа на фото = время получения фото сервером, не время отправки пользователем

### Фиксация всех действий для администратора
- Логировать: пропуски, отклонённые фото, попытки спама, блокировки бота
- В Google Sheets фиксировать статус каждого чек-листа: пройден / пропущен / в процессе
- Пропущенный чек-лист = отдельная строка со статусом "missed" и причиной
- Администратор всегда видит полную картину — даже если сотрудник "ничего не делал"

## Надёжность и защита от саботажа

### 1. Rate limit (anti-spam)
- Не более 10 входящих сообщений в минуту от одного пользователя
- Не более 5 фото в минуту от одного пользователя
- При превышении: отвечать "Слишком часто. Подождите 30 секунд" и игнорировать лишнее
- Счётчики хранить в памяти (Map с TTL), не в БД — для скорости

### 2. Идемпотентность записи в Google Sheets
- Не допускать дублей при повторных попытках отправки
- Ключ идемпотентности: run_id + question_id
- В таблице answers добавить поле sent_to_sheets: boolean
- Если sent_to_sheets = true — повторно не отправлять ни при каких условиях

### 3. Outbox-паттерн для внешних интеграций
- Любая отправка во внешние сервисы (Make webhook) идёт через таблицу outbox в БД
- Схема outbox:
  ```
  id, event_type, payload_json, attempts, next_retry_at, status (pending/sent/failed), last_error
  ```
- Порядок при получении фото:
  1. Сохранить ответ в answers
  2. Добавить событие в outbox со статусом pending
  3. Попытаться отправить немедленно
  4. Обновить статус outbox
- Фоновый воркер (setInterval или node-cron) повторяет отправку pending событий с учётом next_retry_at
- Система работает даже если Make webhook временно недоступен — данные не теряются

### 4. Retry на временные ошибки
- Make webhook: до 3 попыток с exponential backoff (1с → 2с → 4с)
- Повторять при: HTTP 429, HTTP 5xx, сетевые таймауты
- При окончательном fail:
  - outbox.status = failed
  - логировать ошибку с correlation id
  - основной сценарий пользователя НЕ ломать
- AI и Storage: 2 попытки с паузой 1с, при fail — сохранить в БД, продолжить без блокировки

### 5. Таймауты и статусы для пользователя
- Установить таймауты:
  - Storage upload: 15 секунд
  - AI check: 20 секунд
  - Make webhook POST: 10 секунд
- Пользователь всегда получает ответ:
  - Сразу после фото: "Фото принято, идёт проверка..."
  - После проверки: "✅ Шаг засчитан" или "❌ Нужно переснять"
  - При ошибке внешних сервисов: "Фото сохранено, данные будут отправлены позже" — и продолжить
- Никогда не оставлять пользователя без ответа более 30 секунд

### 6. Команда /status для администратора
- Доступна только telegram_id из ADMIN_IDS
- Показывает диагностику кратко:
  ```
  🟢 DB: ok
  🟢 Storage: ok
  🟡 AI: slow (2.3s avg)
  🔴 Make webhook: fail (last error: timeout)
  📬 Outbox: pending 3, failed 1
  ```
- Нужна для быстрого ответа на заявления сотрудников "бот не работает"
- Если всё зелёное — значит бот работает, проблема на стороне сотрудника

### 7. Логи и аудит
- Логировать ключевые события с полями:
  ```
  user_id, role, run_id, question_id, step_state, timing_ms, error_code
  ```
- Все ошибки писать с correlation_id (run_id) для быстрого разбора
- Уровни логов: info (штатные события), warn (нештатные но не критичные), error (сбои)
- Логи хранить в файл с ротацией (winston или pino)

### 8. Защита админ-доступа
- Список администраторов хранить в ADMIN_IDS в .env
- Проверку на админа делать middleware-ом, не в каждом хендлере отдельно
- Неадмин получивший админ-команду — логируется как подозрительное действие

### Главное правило надёжности
Система продолжает работу даже если временно недоступны AI / Make / Storage.
- Данные сначала в БД, потом во внешние сервисы
- Сбой внешнего сервиса = пометка в outbox, догрузка позже
- Пользователь не блокируется из-за внешних сбоев
